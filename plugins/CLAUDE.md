# Plugin Architecture

Three-phase lifecycle: `activate()` inserts services into `AnyMap`, `providers()` returns `Vec<Arc<dyn nyne::router::Provider>>`, and `control_commands()` registers IPC control commands. Registration via `#[linkme::distributed_slice(PLUGINS)]`, linkage auto-generated by `build.rs` from `[dependencies]`.

## Plugin Trait Methods

| Method | Phase | Purpose |
|-|-|-|
| `id(&self)` | — | Unique plugin identifier (config lookup key) |
| `activate(&mut ctx)` | Mutable | Insert shared services into AnyMap |
| `providers(&ctx)` | Immutable | Return middleware chain providers |
| `control_commands(&ctx)` | Immutable | Register IPC control socket commands |
| `scripts(&ctx)` | Immutable | Register named scripts (`nyne exec`) |
| `default_config()` | Static | Default `[plugin.<id>]` TOML table |
| `resolved_config(&config)` | Static | Resolved config for `nyne config` output |

`provider_graph!` and `plugin_config!` macros generate `provider_graph()`, `default_config()`, and `resolved_config()` implementations.

## Plugin Dependency Chain

```
nyne (core)
├── visibility      infrastructure — per-process VFS filtering
├── companion       infrastructure — companion path splitting, @ namespace, extension tree owner
├── cache           infrastructure — readdir cache with generation invalidation
├── slice           infrastructure — line-range path rewriting
├── diff            infrastructure — diff preview-and-apply middleware (.diff files)
├── fs              infrastructure — terminal filesystem provider
├── source          foundational — parsing, decomposition, editing (depends on diff)
│   ├── lsp         progressive — LSP nodes in symbol directories (depends on diff)
│   ├── analysis    progressive — ANALYSIS.md in symbol directories
│   └── git         progressive — per-symbol blame/history
├── nyne            meta — mount status (.nyne.md), no plugin dependencies
├── claude          consumer — hooks use source, lsp, analysis via AnyMap
└── todo            consumer — uses SyntaxRegistry for extension filtering
```

## Invariants

- **No cycles.** Dependencies flow downward only.
- **Core ↛ plugins.** Core has zero knowledge of any plugin. `AnyMap` is the sole inter-plugin communication channel.
- **Source ↛ lsp/analysis/git.** Source has zero imports from downstream plugins.
- **Progressive disclosure.** VFS nodes appear/disappear based on loaded plugins. Multiple providers contribute children to the same path — dispatch auto-merges.
- **Plugin-owned config.** Plugin-specific config lives in `[plugin.<id>]`.
- **Plugin-owned control commands.** Registered via `control_commands()`, not hardcoded.

## Cross-Plugin Communication

| Mechanism | Direction | Example |
|-|-|-|
| Context extension traits | Any plugin → any plugin | `ctx.syntax_registry()` via `SourceContextExt` |
| Request extension traits | Provider → upstream state | `req.companion()` via `CompanionRequest` |
| Chain::evaluate (pipeline) | Any code → middleware state | Hooks call `chain.evaluate(path, op)` |
| Plugin control commands | External → plugin | `nyne ctl` sends `SetVisibility` |

## Writing a New Plugin

1. Create `plugins/<name>/` with `Cargo.toml` (`nyne-plugin-<name>`, lib `nyne_<name>`), `src/lib.rs`, `src/plugin.rs`.
2. Register via `nyne::register_plugin!(<Plugin>)` in `plugin.rs`.
3. `activate()` — insert services via `ctx.insert(value)`, read via context extension traits. For config: implement `PluginConfig`, use `ctx.plugin_config::<Config>(self.id())`. Export services via `<Plugin>ContextExt` in `context.rs`.
4. `providers()` — use `define_provider!` for metadata, `provider_graph!` and `plugin_config!` in `Plugin` impl. Construct providers via struct literals with `pub(crate)` fields.
5. `control_commands()` — register IPC commands.
6. Add workspace dependency alias in root `Cargo.toml`. Linkage auto-generated by `build.rs`.

## Route Extensions

Extension points allow downstream plugins to contribute VFS nodes into upstream route trees without breaching isolation.

1. **Owning plugin** defines a struct with `RouteExtension` fields, inserts into context in `activate()`.
2. **Downstream plugins** call `ctx.<ext>_mut()` during `activate()`, register callbacks via `scoped()`.
3. **Owning plugin** calls `builder.apply(&ext)` in `providers()`.

Callback types: `on_readdir`, `on_lookup`, `on_create`, `content`, `content_always` (additive); `handler` (singular); `dir`, `capture`, `rest` (structural). Extensions carry state via captured `Arc`s. Registration in `activate()`, application in `providers()`. Use `scoped("plugin-name", cb)` for provenance.

`on_create` callbacks materialize write-only sink nodes: attach a `NamedNode` (with `Writable`, plus `CachePolicy::Ttl(d)` declaring the visibility window) to `req.nodes`. The FUSE bridge binds the node to the new inode via `InodeMap::bind_node` — `lookup_node` returns the bound node directly for the `Ttl` window (refreshed on each open/close), then lazily clears on the first access past expiry, restoring sink semantics (`ENOENT` on lookup). Use for endpoints whose presence should stay hidden between writes (e.g. batch-edit `edit/{op}`) but must remain stat-able for a short post-write verification window (e.g. Claude Code's Write tool round-trip).
